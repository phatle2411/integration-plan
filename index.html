Overview

The SmartR integration platform consists of two independent parts:

  • Integration Settings — Organizations create API Keys to grant external services the ability to call into SmartR
  • Integration Marketplace — Admins select and configure pre-supported external services (Jira, BambooHR...) for SmartR to connect to

All authentication uses API Keys only. No OAuth2, no OIDC.

Estimated timeline: 4–5 weeks.

Part 1 — Integration Settings

Purpose

Allows organizations to create and manage SmartR-issued API Keys. These keys are handed to external services (internal tools, scripts, custom systems) so they can:

  • Call SmartR's REST API to read or write data
  • Send webhooks into SmartR to trigger data sync

1.1 — Prisma Schema

Add the following tables to smartr-nest-be:

  • IntegrationApp — Represents an external application that has been granted access. Belongs to an Organization. Stores name, description, and status (active/inactive).
  • ApiKey — Stores the SHA-256 hash of an API Key, linked to an IntegrationApp. Stores assigned scopes, rate limit configuration, and an optional expiry date.

All tables include organizationId as a foreign key — ensuring complete data isolation between organizations.

Admin goes to Integration Settings
        ↓
Creates an Integration App (name, description)
        ↓
Selects allowed scopes
        ↓
SmartR generates an API Key: sk_live_xxxxxxxxxxxxxx
        ↓
Key is displayed ONCE ONLY (cannot be viewed again)
        ↓
Only the SHA-256 hash is stored in the database
        ↓
Admin copies the key and pastes it into the external service

Part 1 — Integration Settings

1.3 — API Key Authentication

When an external service calls into SmartR, it attaches the key to the request header: 

X-API-Key: sk_live_xxxxxxxxxxxxxx

The NestJS ApiKeyGuard will:

  1. Hash the received value using SHA-256
  2. Look it up in the database
  3. On match: attach { organizationId, appId, scopes } to the request context
  4. On no match: return 401 Unauthorized
  5. Redis checks the rate limit — returns 429 Too Many Requests if the threshold is exceeded

Part 2 — Integration Marketplace

Purpose

Allows organization admins to connect SmartR to pre-supported external services. SmartR will actively call out to those services' APIs, or receive webhooks from them, to sync data.

2.1 — Supported Services (V1)

Only pre-defined services supported in code from the start:

Service Integration Type Data Synced
Jira Inbound webhook + Polling Tasks, Projects, Assignees
BambooHR Polling Employees, Departments, Positions
To add a new service in the future: a developer implements a new connector and deploys — admins cannot add services themselves.

Part 2 — Integration Marketplace

2.2 — Prisma Schema

  • ConnectorConfig — Connection configuration for a service per org. Stores: provider (jira/bamboohr), URL, service API Key (AES-256 encrypted), webhook secret, field mapping JSON, sync direction, cron schedule, status, and lastSyncAt timestamp.
  • WebhookEvent — Stores all received inbound webhooks. Stores provider, orgId, event ID, raw payload, and processing status (pending/processed/failed).
  • SyncLog — Result record after each sync run. Stores number of records processed, created, updated, skipped, errors, and timestamp.

Part 2 — Integration Marketplace

2.3 — Admin Flow to Connect a Service

Admin goes to Integration Marketplace
        ↓
Selects a service (e.g. Jira)
        ↓
Fills in connection details:
  - Jira URL (e.g. https://mycompany.atlassian.net)
  - API Key (obtained from their Jira account)
  - Jira account email (used for Basic Auth)
  - Webhook Secret (auto-generated by SmartR, admin copies into Jira)
        ↓
SmartR tests the connection (calls the Jira API to verify credentials)
        ↓
Admin configures field mapping and sync schedule
        ↓
Saves ConnectorConfig (API Key encrypted with AES-256)
        ↓
SmartR displays the Webhook URL for the admin to paste into Jira:
  https://nest.smartrlms.com/api/v1/webhooks/jira/{orgSlug}

Part 2 — Integration Marketplace

2.4 — Inbound Webhook (Jira → SmartR)

When an event occurs in Jira (new task, update...), Jira POSTs to SmartR: 

POST /webhooks/jira/{orgSlug}
        ↓
Verify HMAC signature using webhook secret
        ↓
Check Redis dedup (eventId, 24h TTL) — prevent duplicate processing
        ↓
Save WebhookEvent with status "pending"
        ↓
Enqueue to BullMQ (integration-sync queue)
        ↓
Return 202 Accepted immediately
        ↓
Worker processes asynchronously → Sync Engine

Part 2 — Integration Marketplace

2.5 — Polling (BambooHR and Fallback)

For services that do not support sending webhooks (or when webhooks fail): 

BullMQ repeatable job runs on cron schedule
        ↓
Connector calls BambooHR API using stored key
        ↓
Compare against lastSyncAt — only fetch changed data
        ↓
Feed into Sync Engine
        ↓
Update lastSyncAt

Part 2 — Integration Marketplace

2.6 — Sync Engine

Shared processing pipeline for both webhooks and polling: 

1. Load ConnectorConfig for org + provider
2. Decrypt API Key (AES-256)
3. Connector Registry finds the matching connector (Jira/BambooHR)
4. Connector parses payload → normalized data
5. Field Mapper applies configured mapping
6. Conflict Resolver handles existing entities:
   - external_wins: overwrite SmartR data
   - internal_wins: keep SmartR data unchanged
   - latest_wins: compare timestamps, keep the newest
7. Prisma upsert → persist to database
8. Write SyncLog

Field Mapper — example (configured JSON per connector): external employee fields map onto SmartR tables and columns. job_title can target a related entity (organization_positions), not only the member row.

Part 2 — Integration Marketplace

2.7 — Connector Registry

Each service has a connector class implementing a standard interface:

  • validateCredentials(config) — Test the connection to the service
  • handleWebhookPayload(payload, config) — Parse webhook into normalized data
  • poll(config, since) — Fetch changed data from the API
  • getDefaultFieldMapping() — Return a sensible default field mapping
  • verifySignature(headers, body, secret) — Verify inbound webhook signature

To add a new service: simply implement this class and register it in the Registry — no impact on the rest of the system.
